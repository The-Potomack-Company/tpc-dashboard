<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local dev machine → Supabase CLI (auth token) | Only the human has the token; Claude never sees it. |
| service_role → RPC body | Already inside trust boundary (service_role bypasses RLS). RPC is the interior. |
| JSON payload (future: from parser) → RPC cast machinery | Future untrusted content; casts reject malformed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02 | Tampering (SQL injection via PDF content) | `public.import_sale_with_departments(jsonb, jsonb)` | mitigate | All fields extracted via `->>` with typed casts (`::numeric`, `::int`, `::date`, `::boolean`, `::timestamptz`). No string concatenation. Cast errors abort the transaction → per-sale rollback. |
| T-05 | Tampering (wrong-DB push) | `npm run db:push` | mitigate | Checkpoint task explicitly instructs the human to confirm target via `supabase projects list` / `.env.local` before pushing. |
| T-06 | Information disclosure (logs expose secrets) | Supabase CLI output | accept | `supabase db push` stdout contains no secrets; access token (if used) is in env only. Low risk. |
| T-07 | Elevation of privilege (RPC callable by authenticated users) | Function GRANT | mitigate | `revoke all from public` + `grant execute to service_role` only. Acceptance criteria grep-verifies. Function uses `security definer` with `set search_path = public, pg_temp` to defuse search-path hijack. |
| T-08 | Denial of service (RPC called in a loop by attacker) | Function access | accept | service_role is server-side only; not exposed to authenticated users. No attack surface in v1. |
</threat_model>
