## Pivot Note (added 2026-04-24)

**v1.0 is being archived with its outstanding HUMAN-UAT items accepted, not resolved — because the product direction pivoted before those flows could be validated in a live session.**

### What changed

The original v1.0 thesis was:
> Give the TPC team a single place to see how their auctions are performing over time — what departments are strong, which sales do well — so they can make better decisions about future sales.

Built around that: RFC PDF import (Phase 2), sale browsing (Phase 3), KPI landing (Phase 4), trend charts (Phase 5), department/sale comparison (Phase 6). About 364 of 457 historical PDFs were imported during UAT setup.

**New direction (v2.0 scope, to be planned next):**
1. **TPC App activity tracking** — read-only visibility into the voice cataloger sessions, items, specialist workload, export history.
2. **TPC AI Cataloger extension analytics** — consume the extension v2.0 `analytics_events` stream.
3. **Live RFC sale tracking** — real-time scrape of an active auction (current lot, bids, clearance pace) while the sale is running.

Historical analytics, PDF import, department/sale comparison, and trend charts over historical data are **out of scope** for v2.0.

### What's being done with v1.0 work

| Artifact | Action |
|----------|--------|
| Phase 1 (auth, layout, routing, deploy config) | **Retained** — foundation for v2.0. |
| Phase 2 (PDF import pipeline) | **Retired** — scripts deleted, migrations reverted, DB tables dropped. |
| Phases 3–6 (sale views, KPI, trends, dept/compare) | **Retired** — pages, components, hooks, libs deleted. |
| Imported auction data (~364 sales) | **Purged** from shared Supabase project (truncated + tables dropped). |
| Git history | **Preserved** — all phase 2–6 commits remain on the archived milestone branch for reference if requirements ever return. |
| HUMAN-UAT items (38 total across all 6 phases) | **Status flipped to `accepted_with_debt_pivot`** — the features worked in automated tests but will not get their browser-session validation because the feature is going away. Phase 1's deploy-smoke UAT stays `pending` (deploy still matters for v2). |
| Nyquist gaps (phases 3, 5, 6) | **No action** — won't re-validate features that are being deleted. |
| Code review info items (IN-01..07) | **Dropped** — attached to files being deleted. |

### Lessons captured

The v1.0 build was technically clean: 644/644 tests green, tsc clean, 0 critical / 0 warning code review findings (after WR-01/02/03 fixes). The pivot is a **product decision**, not a code-quality decision. Retiring the code doesn't invalidate the work — it reflects that the product hypothesis was tested at the code-complete boundary and the direction changed before the feature shipped to live users. The RFC PDF parser, dashboard-owned schema design, RLS pattern, and TanStack Query hook pattern all remain valid references for v2.0 even though the specific features are going away.
