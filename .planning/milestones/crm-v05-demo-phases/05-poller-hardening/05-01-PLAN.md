## Plans

### Plan 05-01-a — Reconcile closed-stage boxes (P1)

**Source:** Codex review `review-mpfq2hki-g6jqbw` finding P1 on
`api/lib/crm/streakApi.ts:81-90`.

**Problem:** when a box previously persisted as a row in `crm_threads`
moves out of an open Streak stage (e.g. into a closed/won/lost stage),
the current `listOpenBoxes` filter prevents it from ever being returned
again. `crm-poll.ts` only upserts rows for boxes that come back from
that fetch; `crm_triage_queue` selects all classified `crm_threads`
without an active/closed flag. Net effect: closed consignments stay in
the inbox **forever**.

**Fix shape (one option — finalize during execution):**
- Add a boolean column `is_active` (default `true`) to `crm_threads`.
  *Schema change — needs a migration. Update `_workspace/Schema/schema.md` first.*
- In the poller, build a set of `streak_box_key`s returned by the
  open-stage fetch this run. After upserting, run a single update that
  marks any `crm_threads` row not in that set as `is_active = false`.
- Filter `crm_triage_queue` view to `WHERE is_active = true`.

**Acceptance:**
- Move a real box to a closed stage in Streak; next poll, the row
  disappears from `/crm/inbox`.
- Move it back; next poll, it reappears (idempotent re-activation).
- Unit test in `api/__tests__/crm-poll.test.ts`: two-pass scenario —
  pass 1 returns box A; pass 2 returns empty; assert A flipped to
  `is_active = false`.

**Read-only invariant:** unchanged. This is local DB state, not Streak
mutation.

### Plan 05-01-b — Propagate Streak rate-limit failures (P2)

**Source:** Codex review `review-mpfq2hki-g6jqbw` finding P2 on
`api/lib/crm/streakApi.ts:119-125`.

**Problem:** when a per-stage `boxes` request hits Streak's 429,
`streakFetch` throws `StreakRateLimited`, but the catch block in
`listOpenBoxes` swallows it and returns an empty page for that stage.
`/api/crm-poll` then returns HTTP 200 with partial/missing leads
instead of using the 503/Retry-After path the handler already
implements for rate-limit propagation.

**Fix shape:**
- In `listOpenBoxes` catch (around `streakApi.ts:119-125`), re-throw
  `StreakRateLimited`. Keep the empty-page fallback only for
  genuinely-empty 200 responses and 404s.
- Verify `crm-poll.ts` already maps `StreakRateLimited` to HTTP 503
  with `Retry-After`. If not, wire it.

**Acceptance:**
- Mock Streak returning 429 on one stage; `/api/crm-poll` responds
  with 503 + `Retry-After`, not partial 200.
- Unit test in `api/lib/crm/__tests__/streakApi.test.ts`: 429 on a
  stage request triggers `StreakRateLimited`, caller sees the throw.
- Existing happy-path tests (empty stage = empty result) stay green.

**Read-only invariant:** unchanged.
