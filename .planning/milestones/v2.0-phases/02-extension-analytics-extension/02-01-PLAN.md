   **Implementation pattern — mirror Pattern 2's `bounds` CTE structure verbatim:**
   ```sql
   create or replace function public.get_cancellation_rates(
     p_from timestamptz,
     p_to timestamptz,
     p_users text[] default array[]::text[],
     p_versions text[] default array[]::text[]
   )
   returns table (
     event_type text,
     cancelled_count bigint,
     total_count bigint,
     rate numeric,
     previous_rate numeric
   )
   language sql
   stable
   security invoker
   as $$
     with bounds as (
       -- D-05 — same length, immediately preceding (mirrors get_kpi_totals)
       select
         p_from                               as cur_from,
         p_to                                 as cur_to,
         p_from - (p_to - p_from)             as prev_from,
         p_from                               as prev_to
     ),
     scoped as (
       -- D-01 — every CTE narrows to extension rows
       select event_type, cancelled, created_at
       from public.analytics_events, bounds
       where app_source = 'tpc-extension'
         and event_type in ('catalog_batch', 'portal_upload')
         and (cardinality(p_users)    = 0 or user_email        = any(p_users))
         and (cardinality(p_versions) = 0 or extension_version = any(p_versions))
         and created_at >= bounds.prev_from
         and created_at <  bounds.cur_to
     ),
     cur as (
       select
         s.event_type,
         count(*) filter (where s.cancelled = true)                                     as cancelled_count,
         count(*)                                                                       as total_count
       from scoped s, bounds b
       where s.created_at >= b.cur_from and s.created_at < b.cur_to
       group by s.event_type
     ),
     prev as (
       select
         s.event_type,
         count(*) filter (where s.cancelled = true)::numeric                            as prev_cancelled,
         count(*)::numeric                                                              as prev_total
       from scoped s, bounds b
       where s.created_at >= b.prev_from and s.created_at < b.prev_to
       group by s.event_type
     ),
     -- Make sure both target event types appear in the result even when one period has zero rows.
     types(event_type) as (values ('catalog_batch'), ('portal_upload'))
     select
       t.event_type,
       coalesce(c.cancelled_count, 0)::bigint                                           as cancelled_count,
       coalesce(c.total_count,     0)::bigint                                           as total_count,
       case when coalesce(c.total_count, 0) = 0
            then 0::numeric
            else round(coalesce(c.cancelled_count, 0)::numeric / c.total_count::numeric, 4)
       end                                                                              as rate,
       -- previous_rate uses NULLIF so denominator=0 yields NULL → JS sees "no delta"
       (coalesce(p.prev_cancelled, 0) / nullif(p.prev_total, 0))::numeric               as previous_rate
     from types t
     left join cur  c on c.event_type = t.event_type
     left join prev p on p.event_type = t.event_type;
   $$;

   grant execute on function public.get_cancellation_rates(timestamptz, timestamptz, text[], text[]) to authenticated;
   ```

   Notes:
   - `previous_rate` uses `NULLIF(prev_total, 0)` so divide-by-zero returns SQL NULL → supabase-js delivers `null` to TypeScript → `CancellationRateKpis` (Plan 02-07) sees `previous_rate == null` and omits the delta entirely. This is the documented contract from Pattern 2.
   - `cur` numerator/denominator are bigint (counts); `previous_rate` is numeric (rate). The 4-decimal `round(...)` matches `get_error_rate_by_type` precision.
   - The `types(event_type) as values ...` left-join guarantees BOTH `catalog_batch` and `portal_upload` rows appear even when one or both periods have zero rows in the active filter (otherwise the UI's 2-card slot would silently shrink to 1 card).
