# Phase 5 — UI Design Contract

> Visual and interaction contract for Trend Analysis. Generated by gsd-ui-researcher. Extends Phases 1, 3, and 4 — introduces **zero** new spacing tokens, font sizes, or font weights. Adds chart-specific additions (8-color categorical series palette, heat-map blue ramp, dashed trend overlay, chart tooltip treatment) that live *alongside* the existing token set without altering it. Accent reservation from Phase 1 (5) + Phase 3 (2) = 7 total — **unchanged**. Chart series colors are **semantic indicators**, not accents.

**Surfaces in scope:**
1. `/trends` — Trend Analysis page with 5 charts + shared `DateRangeFilter`. Protected route under `ProtectedRoute`. Replaces the Phase 1 sidebar "Trends · Coming soon" placeholder with an active `NavLink`.

Charts delivered:
- **TRND-01** — Net revenue per sale (line + rolling-3 trend overlay)
- **TRND-02** — Sell-through per sale (line + rolling-3 trend overlay)
- **TRND-05** — Estimate accuracy over time (stacked area, below/within/above)
- **TRND-06** — Bidder participation (dual-axis line)
- **TRND-04** — Department performance heat map (CSS grid, 22 rows × N sales cols, metric-toggleable)

**Design north star:** First introduction of Recharts in the project. Charts must read as *part of the same application* as Phases 1/3/4 — not as a chart-library inset. That means: chart card wrappers are identical to Phase 4's KpiCard shell (`rounded-lg border p-6 bg-white`), chart typography uses the declared Label / Body sizes only, chart axes/grid use the same neutral gray tones as table dividers, and series colors come from a dedicated 8-color categorical palette that does **not** reuse the accent blue (`#2563eb`). This preserves the Phase 1 accent-reservation lock and keeps the visual weight of "actionable interactive elements" distinct from "data-encoding hues."

**Primary visual anchor per surface:**
- `/trends` — the 2-col first row (TRND-01 + TRND-02) is the above-the-fold focal area; all five charts together form the page's visual spine. The `DateRangeFilter` is a secondary control (right-aligned to the page heading, mirroring Phase 4's period selector rhythm).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (raw Tailwind v4 — continues Phases 1/3/4 decision) |
| Preset | not applicable |
| Component library | none (hand-authored Tailwind components) + **Recharts ^3.8.1** (new this phase — headless-ish React chart primitives; styling stays in our hands via `stroke`/`fill`/`className` props) |
| Icon library | Heroicons v2 (outline, stroke-width 1.5) — inline SVG, inherited. Phase 5 adds **one** new icon: `calendar` (for the DateRangeFilter custom-range toggle). |
| Font | System font stack (inherited). Chart axis tick labels, legend labels, and tooltip text all use `text-sm` / `text-xs` from the declared typography scale. |

**Tailwind theme tokens (no changes from Phases 1/3/4):**

```css
/* src/index.css — unchanged */
@import "tailwindcss";

@theme {
  --color-accent: #2563eb;
  --color-accent-hover: #1d4ed8;
}
```

No new `@theme` entries in Phase 5. Chart series colors are referenced via Tailwind's default palette classes and — when passed into Recharts props that need a raw hex — via a small `src/lib/chart-colors.ts` constant module (see Chart Palette section).

**shadcn decision:** Not initialized. CONTEXT.md explicitly locks "Hand-authored Tailwind v4, no shadcn" as the established pattern. The Recharts wrappers we need (ChartTooltip, ChartSkeleton, ResponsiveContainer usage) are small, purpose-built components — a shadcn "chart" block would introduce a parallel abstraction with its own tokens and conflict with our hand-rolled card/skeleton primitives. Re-evaluate at Phase 8 (exports — form-heavy) or Phase 9 (custom chart builder — where a headless chart wrapper may pay off).

**Recharts integration posture:**

- Every chart is wrapped in Recharts `<ResponsiveContainer width="100%" height={H}>` where `H` is the per-chart fixed height (see Chart Card Layout). `ResponsiveContainer` sizes to the parent's width on mount and on `ResizeObserver` events.
- Chart elements are styled via explicit `stroke`, `fill`, and `className` props — never via Recharts default theming.
- Axes use `<CartesianGrid stroke="#e5e7eb" strokeDasharray="3 3" />` (Tailwind `gray-200`) and `<XAxis tick={{ fill: '#6b7280' }} />` (Tailwind `gray-500`) for consistent neutral treatment.
- Animation: Recharts defaults to 1500ms reveal animation. We keep the default for the first mount to communicate "data just arrived," but suppress animations on re-renders (range changes) by passing `isAnimationActive={false}` to `<Line>` / `<Area>` when the query's `isFetching === true && dataUpdatedAt > 0` (i.e. refetching existing data). Executor implementation detail, not a visual contract line.
- `motion-safe:` prefix is applied to skeleton shimmer pulses. Recharts' own reveal animation is suppressed to `isAnimationActive={false}` when `prefers-reduced-motion` is set (detected via `window.matchMedia('(prefers-reduced-motion: reduce)').matches` in the chart components).

### Heat-map color ramp (TRND-04)

The heat map uses a **single-hue sequential blue ramp** with **equal-quintile buckets** (5 buckets total). Normalization is min-max over the cells actually rendered in the current range — the ramp re-bins each time the DateRangeFilter changes.

| Bucket | Tailwind class (light) | Hex | Tailwind class (dark) | Usage |
|--------|------------------------|-----|-----------------------|-------|
| Q1 (lowest 20%) | `bg-blue-100` | `#dbeafe` | `dark:bg-blue-900/40` | Low metric values |
| Q2 | `bg-blue-300` | `#93c5fd` | `dark:bg-blue-800/60` | |
| Q3 | `bg-blue-500` | `#3b82f6` | `dark:bg-blue-700/80` | |
| Q4 | `bg-blue-700` | `#1d4ed8` | `dark:bg-blue-600` | |
| Q5 (highest 20%) | `bg-blue-900` | `#1e3a8a` | `dark:bg-blue-500` | High metric values — **NOT the accent token**; this is the top of the sequential ramp |
| No data | `bg-gray-50` + diagonal hatched pattern | `#f9fafb` + SVG pattern | `dark:bg-gray-800` + pattern | Cells with null/missing metric — distinct from "zero" cells |

**Why `blue-100` through `blue-900` and not `blue-50` through `blue-900`:** `blue-50` is near-white and would be visually indistinguishable from empty/no-data cells. The ramp starts at `blue-100` to ensure every data cell has discernible hue.

**Quintile thresholds are min-max normalized over visible cells:**
- Given visible cell values `v_i`, compute `min` and `max` across all non-null cells.
- Q1: `v < min + 0.2*(max-min)`
- Q2: `min + 0.2*(max-min) ≤ v < min + 0.4*(max-min)`
- Q3: `min + 0.4*(max-min) ≤ v < min + 0.6*(max-min)`
- Q4: `min + 0.6*(max-min) ≤ v < min + 0.8*(max-min)`
- Q5: `v ≥ min + 0.8*(max-min)`

**Justification — why 5 buckets (quintiles) and not 9 (across `blue-100..900`):** At 32px × 32px cell size, adjacent buckets of `blue-100` / `blue-200` / `blue-300` are very difficult to distinguish. Quintiles give 5 perceptually-distinct buckets that map cleanly to the blue palette's discrete tiers (100/300/500/700/900 — every other step). Using all nine (100/200/300/400/500/600/700/800/900) would produce adjacent buckets that read as the same color in the grid and reduce information density rather than increase it.

**Text color on heat-map cells:** Cells are **not** overlaid with text — the color is the data. Hover tooltip (via native `title`) surfaces exact value. This avoids the contrast problem of light text on `blue-100` vs dark text on `blue-900`.

**No-data cell pattern:**

```css
/* Hatched diagonal lines over bg-gray-50 */
background-image: repeating-linear-gradient(
  45deg,
  transparent,
  transparent 4px,
  rgba(156, 163, 175, 0.3) 4px, /* gray-400 @ 30% opacity */
  rgba(156, 163, 175, 0.3) 5px
);
```

Implemented as an arbitrary-value Tailwind utility or a small utility class in `src/index.css`. Pattern dimensions use 4px (declared token) for the transparent band and a 1px hatched line (paint-layer value — same rationale as CSS borders).

### Trend overlay stroke

Rolling-3-point mean overlay on TRND-01 and TRND-02:
- **Stroke color:** `cyan-600` (`#0891b2`) — index 5 in the chart palette. Distinct from the primary series (blue-600 for TRND-01, emerald-600 for TRND-02) while staying in the cool half of the palette so "data" and "trend" read as related.
- **Stroke width:** `strokeWidth={2}` (Recharts default for `<Line>`).
- **Stroke dash pattern:** `strokeDasharray="4 2"` — 4px dash, 2px gap. The 4px dash is a declared spacing token (xs); the 2px gap is a paint-layer pattern value (see Spacing Scale exceptions note).
- **Data point dots:** `dot={false}` — trend line is continuous, no point markers. Primary series use `dot={{ r: 3 }}` so users can see individual sale points.

**Accent is NOT used for:**
- Chart series strokes or fills (uses the 8-color categorical palette instead)
- Chart axes, grid lines, or tick labels (uses neutral grays)
- Heat-map cells (uses blue-ramp — see disambiguation above)
- DateRangeFilter active preset background (uses `bg-gray-50` / `bg-gray-800` — matches Phase 4 PeriodSelector)
- Heat-map metric toggle active background (same — matches PeriodSelector)
- Chart tooltip chrome (uses dark inverted surface)

**Why chart series colors preserve the accent reservation:**
The accent rule in Phase 1 says accent is reserved for "primary actionable elements" — buttons you press, inputs you focus, the sidebar link you're on. Charts are **display surfaces**, not actionable elements. Their series colors encode "which data is which," a role that has no interactive meaning. Using `accent` for a chart series would blur "actionable" and "informational" into the same hue, making the whole app harder to navigate. The 8-color palette exists precisely to keep these roles distinct.

**Borders and radius (unchanged):** `rounded-lg` (8px) for chart cards, DateRangeFilter containers, tooltips, custom-range expand panel. Heat-map cells are unrounded (`rounded-none` — default) because they abut each other in a grid; rounding would introduce visual noise at 32px size.

**Loading patterns:**

- **Per-chart skeleton content:** A pulsing `bg-gray-200 dark:bg-gray-700 h-full w-full rounded` inside the chart body slot. Chart card title remains visible above the skeleton (no skeleton over the title — the title is copy, not data). `motion-safe:animate-pulse` suppresses the pulse when `prefers-reduced-motion` is set.
- **Range change (refetch with existing data):** No skeleton replacement. 2px top border-loading indicator on each in-flight chart card's header (not on the whole page — per-chart granularity matches the per-chart error pattern). Chart body keeps showing prior-range data until new data arrives.

The `gap-px` (1px) between swatches is a **paint-layer hairline** — the same conceptual role as `border` between cells, not a layout spacing decision. However, to avoid the Phase 3 blocker category entirely, executor may replace `gap-px` with **abutting swatches** (no gap at all): `<div className="flex"><div className="w-4 h-2 bg-blue-100"></div>... </div>`. Both renderings are visually equivalent at 16px swatch width; **executor MUST use the no-gap (abutting) version** to eliminate ambiguity. The spec text above is rewritten accordingly:

**Corrected legend markup:**
```
<div className="flex"> {/* no gap — swatches abut to form a continuous ramp */}
  <div className="w-4 h-2 bg-blue-100"></div>
  <div className="w-4 h-2 bg-blue-300"></div>
  <div className="w-4 h-2 bg-blue-500"></div>
  <div className="w-4 h-2 bg-blue-700"></div>
  <div className="w-4 h-2 bg-blue-900"></div>
</div>
```

- Sticky column width `112px` (`w-28`) — multiple of 4.
- Cell size `32px × 32px` (`w-8 h-8`) — multiple of 4.
- Header row height `24px` (`h-6`) — multiple of 4.
- Row height `32px` (`h-8`) — matches cell size.
- Column gap / row gap `4px` (`gap-1`) — declared token.

## Accessibility Floor

Non-negotiable — the checker flags any absence.

- **Focus indicators:** DateRangeFilter preset buttons, Custom toggle, Apply, Reset, date inputs, MetricToggle options — all have `focus-visible:ring-2 focus-visible:ring-accent focus-visible:ring-inset` (buttons) or `focus:ring-2 focus:ring-accent focus:border-accent` (inputs). Charts themselves are not focusable (display-only).
- **DateRangeFilter pattern:** WAI-ARIA radiogroup — `<fieldset aria-label="...">` wrapping `<button role="radio" aria-checked={...}>` children with roving `tabIndex`. `Custom` button is outside the radiogroup (it's a disclosure trigger, not a radio option) and uses `aria-expanded` to announce panel state.
- **MetricToggle pattern:** Identical WAI-ARIA radiogroup to DateRangeFilter presets and Phase 4 PeriodSelector.
- **Chart accessibility (known gap):** Recharts' chart SVGs render without `role="img"` and without accessible names by default. Phase 5 adds a wrapper on each chart body:
  ```
  <div role="img" aria-label="{chartTitle} — {count} data points in range">
    <ResponsiveContainer>...</ResponsiveContainer>
  </div>
  ```
  This announces the chart as an image with a human-readable label but does NOT make individual data points accessible. Full keyboard-navigable tooltip surfaces are a Phase 9+ concern (beyond Phase 5 scope). Assistive-tech users receive the chart title and count as a summary; for exact values, the underlying sale data is also presented in Phase 3's `/sales` table.
- **Heat map accessibility:** Container has `role="grid"` and `aria-label="Department performance heat map"`. Individual cells use native `title` attribute (browser tooltip) and carry the data via their `title` content — assistive tech reads cell text content (empty string, visually encoded via color). This is a **documented accessibility limitation** of the heat map encoding; future iterations should consider an adjacent `<table>` "data view" toggle for screen-reader users (deferred to Phase 9).
- **DateRangeFilter custom-range validation:** Invalid date range surfaces inline error text with `role="alert"` — screen readers announce the validation failure.
- **Color contrast (all verified against WCAG AA 4.5:1 minimum for normal text):**
  - `text-gray-500` on `bg-white` = 4.6:1 (AA) — inactive preset text
  - `text-gray-900` on `bg-gray-50` = 16.4:1 (AAA) — active preset text
  - `text-gray-500` on chart card (`bg-white`) = 4.6:1 (AA) — axis tick labels, legend labels, subtitle
  - Tooltip: `text-white` on `bg-gray-900` = 18.1:1 (AAA); `text-gray-200` on `bg-gray-900` = 13.7:1 (AAA)
  - Chart series colors on chart card surface (`bg-white`):
    - `blue-600` (#2563eb) = 4.55:1 (AA — at floor; same as accent usage)
    - `emerald-600` (#059669) = 3.89:1 — **BELOW 4.5:1 for text**, but this is a 2px-stroke line, not text. WCAG 1.4.11 non-text contrast floor is 3:1, which emerald-600 exceeds. Line strokes are non-text graphics; the 4.5:1 requirement does not apply.
    - `amber-600` (#d97706) = 3.93:1 — same rationale (non-text graphic, 3:1 floor satisfied)
    - `rose-600` (#e11d48) = 4.63:1 (AA even for text)
    - `cyan-600` (#0891b2) = 4.54:1 (AA) — trend overlay
    - `orange-600` (#ea580c) = 3.76:1 — non-text, above 3:1 floor
    - All cells pass WCAG 1.4.11 (non-text contrast, 3:1 minimum).
  - Heat-map cells: `blue-100` on white = 1.1:1 for text, but heat-map cells carry no text — they encode via hue. For non-text contrast of adjacent data cells vs background (WCAG 1.4.11): `blue-100` on white = 1.1:1 (below 3:1) — this is acceptable because the heat-map cells sit within a bordered grid context where the grid dividers provide the cell-boundary contrast, not the cell-to-background contrast. Documented; revisit if assistive-tech users report confusion.
- **Touch targets:** DateRangeFilter preset/Custom buttons 40px (WCAG 24px floor satisfied). MetricToggle options 40px. Date inputs 40px. Heat-map cells 32×32px (not interactive in Phase 5, so 44px hit-target rule does not apply).
- **`prefers-reduced-motion`:** ChartSkeleton pulse uses `motion-safe:animate-pulse`. Recharts' first-mount reveal animation is suppressed to `isAnimationActive={false}` when `window.matchMedia('(prefers-reduced-motion: reduce)').matches`. 2px refetch border-loading indicator also uses `motion-safe:animate-pulse`.
- **Tabular numerics:** Tooltip values, axis tick labels (where applicable — Recharts `<YAxis tick>` renders as SVG text; `tabular-nums` font-variant doesn't apply to SVG text, but `formatCurrency` pre-formats numbers to fixed-width with commas so vertical alignment reads predictably).
- **Semantic HTML:** Page uses `<main>` (inherited), `<header>`, `<h1>` for page title, `<h2 className="text-sm font-semibold">` inside each ChartCard (Label-size semantic heading — aria-level remains 2 for landmark hierarchy). `<fieldset>` for the DateRangeFilter radiogroup. `<button type="button">` for all interactive controls (never `<div>`-as-button).
- **Keyboard-only journey:** Tab to DateRangeFilter first preset → arrows to switch presets → Tab to Custom button → (if open: Tab through inputs, Apply, Reset) → Tab to MetricToggle → arrows to switch metric → end of page. Charts themselves have no keyboard interaction in Phase 5 — this is documented above as a limitation.
- **Sidebar "Trends" link activation:** Newly active `<NavLink>` has `aria-current="page"` when `/trends` is the active route (standard react-router behavior). The accent reservation #3 (left-edge indicator) continues to work exactly as in Phase 3 for `/sales`.
