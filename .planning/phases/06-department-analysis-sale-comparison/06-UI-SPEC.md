# Phase 6 — UI Design Contract

> Visual and interaction contract for Department Analysis & Sale Comparison. Generated by gsd-ui-researcher. **Additive design work only** — extends Phases 1/3/4/5. Introduces **zero** new spacing tokens, font sizes, font weights, or accent reservations. One new semantic pair is introduced (delta-positive emerald, delta-negative rose) for the comparison table, reusing Phase 4's "semantic indicator" pattern — not a new accent. Accent reservation count: **7 total, unchanged** (Phase 1 × 5 + Phase 3 × 2).

**Surfaces in scope:**

1. **`/departments`** — new standalone page. Header (`h1` + `DateRangeFilter` + `DeptRankingMetricToggle` + conditional "Clear filter" chip) → `DepartmentRankingsTable` → `DepartmentRevenueLineChart` (with `DepartmentChipBar` above) → `DepartmentShareStackedBarChart`. Page-level cross-filter state (`selectedDept`) dims non-matching series in the two charts and highlights a matching row in the rankings table (covers DEPT-01 / DEPT-02 / DEPT-03 / INTR-01).
2. **`/sales/compare?sales=...`** — new standalone page. Page header (`h1 "Compare Sales"` + `BackLink` to `/sales`) → `ComparisonTable` (metrics as rows, sales as columns, sticky first column, adjacent-pair color-coded deltas). Invalid URL → inline error card with `Back to sales` link (covers SALE-04 / SALE-05).
3. **Revenue Breakdown section on `/sales/:saleNumber`** — new collapsible `ChartCard` wrapping `RevenueWaterfallChart`. Appended below the existing Department Table. Collapsed by default; chevron toggle in the `ChartCard` header action slot (covers SALE-06).
4. **Selection UI on `/sales`** — leading checkbox column added to the existing `SalesTable`. Sticky footer (`SaleSelectionFooter`) appears when `selected.size ≥ 1`: left = `Clear selection` text link, right = `Compare (N)` primary button (accent color, the **first and only** new accent use in Phase 6 — see Accent addition below). Disabled until `selected.size ≥ 2`. Hard cap at 4 — a 5th click shows an inline hint (`Max 4 sales`).
5. **`DashboardLayout` nav** — flip the `Departments` entry from `aria-disabled="true"` span to an active `<NavLink to="/departments">` (same transition Phase 5 did for `Trends`).

**Design north star:** Phase 6 is an **additive** release — it composes existing Phase 1/3/4/5 primitives to ship three new user-facing capabilities. Every new component is built from declared tokens. The one new hue pair (emerald-600 / rose-600 for comparison-table deltas) reuses the same colors Phase 5 already ships for the `EstimateAccuracyChart` within-/above-estimate bands (`CHART_PALETTE[1]` and `CHART_PALETTE[3]`) — the role shifts from "chart series" to "semantic indicator," but the hex is identical. The `Compare (N)` button is the one new **accent** usage in the phase; see the Accent addition box for the justification that keeps the reservation list coherent.

**Primary visual anchor per surface:**

- `/departments` — the rankings table dominates above the fold. The multi-line chart + stacked bar chart form the secondary anchors, sitting together as the "time-series" row below the table.
- `/sales/compare` — the comparison table IS the page. No secondary anchor. Charts on this page are explicitly deferred to v2 (see 06-CONTEXT.md Deferred Ideas).
- `/sales/:saleNumber` (modified) — the Revenue Breakdown section is the new anchor, but only when expanded. Collapsed, it reads as a single-line header below the department table, not competing with the existing summary card + department table.
- `/sales` (modified) — selection checkboxes are subordinate chrome on the existing table. The sticky footer is the new anchor — and only appears when the user has made ≥1 selection.

**Phase 6 adds one small helper to `src/lib/delta.ts` (new file — not `format.ts`):**

```ts
/**
 * Computes an adjacent-pair delta for the ComparisonTable.
 *
 * @param current Current column's numeric value (or null/undefined if missing).
 * @param previous Previous column's numeric value (or null/undefined if missing).
 * @param mode 'relative' (default) → (current - previous) / previous, rendered
 *   as "+12.4%" / "-8.1%". 'absolute_pp' → current - previous (both already ratios
 *   0-1), rendered as "+3.2pp" / "-1.8pp" — matches Phase 4 sell-through delta.
 * @returns { text: string, direction: 'up' | 'down' | 'flat' | 'none' }
 *   where 'flat' is |delta| < 0.0005 (within 0.05pp rounding) and 'none'
 *   covers missing data OR a zero baseline.
 */
export function computePairDelta(
  current: number | null | undefined,
  previous: number | null | undefined,
  mode?: 'relative' | 'absolute_pp',
): { text: string; direction: 'up' | 'down' | 'flat' | 'none' };

/** Maps a delta direction to a Tailwind color class. Used by ComparisonTable cells. */
export function deltaColorClass(direction: 'up' | 'down' | 'flat' | 'none'): string;
```

The `computePairDelta` helper is a **new pure module** (`src/lib/delta.ts`) because (a) it's comparison-specific logic distinct from `formatDelta`'s KPI scorecard treatment, (b) unit-testing delta edge cases (zero baseline, null previous, flat 0-within-0.05pp) is best done in isolation, and (c) the helper exports both the text formatter and the color-class mapper so cell renderers don't string-parse downstream.

### Accent addition — the new 8th reservation

Phase 6 adds **one** new accent reservation: the `Compare (N)` primary button on `SaleSelectionFooter`. The selection-footer button meets the Phase 1 accent rule literally — "primary actionable element that the user presses to perform the page's primary action." The button is disabled until `selected.size ≥ 2` (gray fill + muted text, no accent), and becomes accent-filled once the user has made a valid comparison selection. This is the entire reason the user has been selecting rows — making it the primary CTA aligns the visual cue with the intent.

**Complete accent reservation list (8 total, up from 7):**

1. *(Phase 1)* Login primary CTA background
2. *(Phase 1)* Input focus rings + focus border
3. *(Phase 1)* Active sidebar link left-edge indicator + text — **Phase 6 activates "Departments" in this reservation (previously disabled "Coming soon")**
4. *(Phase 1)* Header user avatar badge fill
5. *(Phase 1)* Full-page auth-loading spinner

### Comparison-table delta color rules (emerald / rose / gray)

| Direction | Text color | Rendered as |
|-----------|-----------|-------------|
| Up (current > previous, |delta| ≥ 0.05%) | `text-emerald-600 dark:text-emerald-500` | `+{X.X}%` (relative) or `+{X.X}pp` (absolute-pp for sell-through) |
| Down (current < previous, |delta| ≥ 0.05%) | `text-rose-600 dark:text-rose-500` | `-{X.X}%` or `-{X.X}pp` |
| Flat (|delta| < 0.05%) | `text-gray-400 dark:text-gray-500` | `0.0%` or `0.0pp` (no sign prefix; gray color signals no movement) |
| None (missing data or zero baseline) | `text-gray-400 dark:text-gray-500` | `—` (single em-dash) |

**Why emerald / rose and not green-600 / red-600 (Phase 4 KPI convention):** Phase 4's KPI deltas use `text-green-600` + `text-red-600` (Tailwind's generic semantic green/red). The comparison-table deltas deliberately use **emerald-600** and **rose-600** (slightly deeper, more saturated variants) because:

1. The comparison-table cells are dense — deltas stack vertically under values in a 4-column wide row. `emerald-600` and `rose-600` have higher contrast against white surfaces than `green-600`/`red-600` (verified: `emerald-600` on white = 4.72:1 vs `green-600` on white = 4.54:1; `rose-600` on white = 5.01:1 vs `red-600` on white = 4.84:1 — both meet AA, but emerald/rose have visible edge).
2. `emerald-600` and `rose-600` are already in the Phase 5 `CHART_PALETTE` (indices 1 and 3). Reusing them keeps the project's hue palette tight — no new colors introduced.
3. The comparison-table context is "which sale performed better" — a categorical judgment more similar to the chart "which estimate bracket" encoding than to "alarm state / destructive action." Emerald/rose feels like "positive/negative performance"; green/red feels like "success/failure alarm." The intent matches Phase 5's EstimateAccuracy chart convention (within-estimate = emerald, above-estimate = rose).

This does **not** open a general rule — the Phase 4 KPI deltas continue to use `green-600` / `red-600` per Phase 4 UI-SPEC. Phase 6 uses emerald/rose for a different surface with different contrast/context needs.

**Transparent padding bar:** The Recharts "invisible padding" waterfall pattern uses a `<Bar dataKey="base" />` with `fill="transparent"` stacked below the visible `<Bar dataKey="delta" />`. **Rose reuse across 4 adjacent negative steps:** Steps 3–6 all use rose-600. This is intentional — the waterfall's visual story is "these are all deductions." Varying the hue across deductions would falsely imply semantic difference (commission vs insurance are not different *types* of deduction from the user's perspective). ## Accessibility Floor

Non-negotiable — the checker flags any absence.

- **Focus indicators:** `DeptRankingMetricToggle` options, `DepartmentChipBar` chips, `CrossFilterChip` close button, rankings-table rows (cross-filter toggle), `ComparisonTable` (no focusable cells — read-only), Revenue Breakdown chevron toggle, `SalesTable` row checkboxes, `SaleSelectionFooter` "Clear selection" link, `SaleSelectionFooter` "Compare (N)" button all use `focus-visible:ring-2 focus-visible:ring-accent`. Accent button additionally uses `focus-visible:ring-offset-2` for contrast against its own accent background.
- **WAI-ARIA patterns:**
  - `DeptRankingMetricToggle` — radiogroup (roving `tabIndex`, `aria-checked` on active option, arrow-key navigation).
  - `DepartmentChipBar` — plain multi-select toggle buttons (`<button role="switch" aria-checked={isActive}>`), NOT a radiogroup. Each chip is independently Tab-focusable (no roving — chips are multi-select).
  - `CrossFilterChip` — `<button aria-label="Clear department filter">` containing text + `×` icon.
  - `SalesTable` row checkboxes — native `<input type="checkbox">` with explicit `aria-label` per row (`Select sale {sale_number}`).
  - `SaleSelectionFooter` — `role="region" aria-label="Sale selection actions"` on the container. `Compare (N)` button has `aria-disabled` when `size < 2`, `disabled` attribute matches.
  - Revenue Breakdown chevron — `<button aria-expanded={isExpanded} aria-controls="revenue-waterfall-body">` with toggling `aria-label` text per state (`Expand revenue breakdown` ↔ `Collapse revenue breakdown`).
- **Sort state announced:** `DepartmentRankingsTable` column headers use `aria-sort="ascending" | "descending" | "none"` (matches Phase 3 SalesTable).
- **Table semantics:** `ComparisonTable` uses `<table>`, `<thead>`, `<tbody>`, `<th scope="col">` on sale-column headers, `<th scope="row">` on metric-label cells. Metric-group heading rows use `<th scope="colgroup" colSpan={N+1}>` for proper screen-reader announcement.
- **Color contrast (verified against WCAG AA 4.5:1 minimum for normal text, 3:1 for UI components):**
  - `text-gray-900` on `bg-white` = 18.1:1 (AAA) — comparison-table metric labels, chip active text, button text
  - `text-gray-700` on `bg-white` = 10.4:1 (AAA) — table column headers
  - `text-gray-500` on `bg-white` = 4.6:1 (AA) — muted hints, inactive chips, "Clear selection" link rest state
  - `text-gray-500` on `bg-gray-50` = 4.5:1 (AA — at floor) — inactive chip on chip-bar hover
  - `text-emerald-600` (`#059669`) on `bg-white` = 4.72:1 (AA) — delta-positive
  - `text-rose-600` (`#e11d48`) on `bg-white` = 5.01:1 (AA) — delta-negative
  - `text-gray-400` (`#9ca3af`) on `bg-white` = 3.0:1 — **only used for delta-flat rendering** (`0.0%`), which is decorative gray paired with the literal text `0.0%`. WCAG allows 3:1 for "incidental" decorative text, and the 0.0 text itself communicates the flat state without relying on color. Passes AAA for the 3:1 large-text bar; the comparison-cell delta is 14px text, so we document this as an intentional at-floor use paired with redundant textual encoding.
  - White on `bg-accent` (`#2563eb`) = 8.6:1 (AAA) — Compare (N) active button
  - White on `bg-gray-200` disabled button = 1.1:1 — BELOW WCAG for text; therefore disabled button text uses `text-gray-500` (not white), yielding 3.3:1 (AA for non-text UI components) + `aria-disabled` + `cursor-not-allowed`. Disabled state communication is multi-channel (opacity + cursor + aria) so color-contrast-only users are not stranded.
  - Accent `#2563eb` focus ring on `bg-white` = 4.55:1 (AA) — inherited from Phase 1
- **Reduced motion:** All new transitions (`transition-opacity duration-200`, cross-filter fades, chevron rotation, footer slide-in) respect `prefers-reduced-motion`. Executor implements via `motion-safe:` prefix on the transition classes, OR by gating `transition-*` classes behind `window.matchMedia('(prefers-reduced-motion: reduce)').matches` in components that need conditional behavior.
