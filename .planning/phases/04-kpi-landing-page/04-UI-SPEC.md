# Phase 4 — UI Design Contract

> Visual and interaction contract for the KPI Landing Page. Generated by gsd-ui-researcher. Extends Phases 1 and 3 — introduces zero new spacing tokens, font sizes, or font weights. Accent reservation from Phase 1 (5 elements) + Phase 3 (2 elements) remains intact; delta indicators use semantic grays/greens/reds, NOT accent.

**Surfaces in scope:**
1. `/` (Dashboard landing) — 4 KPI scorecards + period selector + Recent Sales panel. Replaces the Phase 1 `Your KPIs land here.` placeholder body in `src/pages/Dashboard.tsx`. Route wiring is unchanged.

**Design north star:** The landing page is the user's first post-login view and must read as a *snapshot*, not a dashboard-builder surface. KPI cards take up the top fold; Recent Sales gives immediate drill-down paths. No new visual vocabulary — everything is built from the Phase 1/3 tokens already in the project. Deltas are semantic indicators (green/red/gray) to preserve Phase 1's accent reservation.

**Primary visual anchor:** The 4-card KPI row at the top of the content area. The period selector is a secondary control (right-aligned to the page heading). Recent Sales sits below and acts as a secondary anchor for drill-down.

**Iconography for delta indicators:**

Delta glyphs use Unicode characters, NOT Heroicons. Rationale: a single-character inline glyph renders identically across font smoothing settings, avoids an icon-component indirection for a three-state display, and matches the CONTEXT.md-specified rendering verbatim (`▲ 12.4%`, `▼ 8.1%`, `—`).

| State | Glyph | Codepoint |
|-------|-------|-----------|
| Positive delta | `▲` | U+25B2 BLACK UP-POINTING TRIANGLE |
| Negative delta | `▼` | U+25BC BLACK DOWN-POINTING TRIANGLE |
| No baseline (previous period had 0 sales) | `—` | U+2014 EM DASH (same as `EMPTY` in `format.ts`) |

The glyph and percentage share a single inline span, separated by a normal space: `▲ 12.4%`. Color is applied to the entire span, not just the glyph (glyph and number must read as one unit — color splitting would add visual noise). The span has `aria-label` copy described in Accessibility Floor so screen readers announce the change direction without relying on the visual glyph.

Phase 4 adds **one** new helper to `src/lib/format.ts`:

```ts
/**
 * Formats a period-over-period ratio as `"▲ 12.4%"` / `"▼ 8.1%"` / `"—"`.
 *
 * @param ratio Current/previous ratio expressed as a signed delta
 *   (e.g. 0.124 for +12.4%, -0.081 for -8.1%). Pass `null` when
 *   previous-period baseline is zero/absent.
 * @returns `{ glyph, text }` so the caller can color the span as a whole.
 *   For the null case, returns `{ glyph: '—', text: '' }` — the caller
 *   renders just the em-dash.
 */
export function formatDelta(ratio: number | null | undefined): {
  glyph: '▲' | '▼' | '—';
  text: string;
};
```

The helper returns a structured object (not a single string) so the KpiCard can apply the color class to the full `{glyph} {text}` span atomically without string-parsing downstream.

**Accent reservation (7 total — unchanged; Phase 4 adds none):**

1. *(Phase 1)* Login primary CTA background
2. *(Phase 1)* Input focus rings + focus border
3. *(Phase 1)* Active sidebar link left-edge indicator + text
4. *(Phase 1)* Header user avatar badge fill
5. *(Phase 1)* Full-page auth-loading spinner
6. *(Phase 3)* Active sort column indicator
7. *(Phase 3)* Table / card row focus ring — Phase 4 reuses this reservation for Recent Sales cards (keyboard-navigable, click-to-navigate); period-selector buttons reuse reservation #2 (input-family focus ring)

**Accent is NOT used for:**
- KPI card label, value, or delta text (delta uses semantic green/red/gray)
- KPI card borders or surface
- Period selector active-option background (uses `bg-gray-50` / `bg-gray-800`)
- Period selector active-option text (uses `text-gray-900` / `text-gray-100`)
- Recent Sales card hover background (uses `bg-gray-50` / `bg-gray-800`)
- Recent Sales card text (titles, numbers, etc.)
- Any delta glyph

**Why period-selector active state uses semantic gray (not accent):** The brief explicitly leans toward "no-new-accent — semantic gray." Rationale: the period selector is persistently visible and changing period is a routine action, not a highlighted primary CTA. Using accent would (a) add an 8th accent reservation that violates the Phase 1/3 lock, (b) compete visually with Recent Sales card focus rings which *must* use accent for WCAG keyboard visibility. Gray-based active state (weight + subtle surface fill) communicates selection unambiguously without burning the accent reservation. Contrast check: `text-gray-900` on `bg-gray-50` = 16.4:1 (WCAG AAA).

**Why negative deltas use the existing `red-600` destructive token:** Adding a new "negative trend" color would introduce a 6th semantic role (accent, destructive, warning amber, muted, positive-green, negative-red). Reusing `red-600` keeps the palette tight while matching user convention. The distinction is contextual: destructive appears on retry buttons and error text (alongside copy like "Couldn't load"), whereas delta-negative appears next to a percentage with a `▼` glyph — users parse context before color.

**Borders and radius (unchanged):** `rounded-lg` (8px) for KPI cards, Recent Sales cards, period selector outer container, period selector individual option buttons (no — see below), retry button.

**Period selector border radius treatment:** The outer container is `rounded-lg` (8px). Individual option buttons inside are **not** independently rounded — they're flat-edged because they abut each other to form the segmented control. The first option's left edge and last option's right edge inherit the container's rounded corners via `overflow-hidden` on the outer container. This is the standard segmented-control pattern and introduces no new radius token.

### KPI scorecards

Card labels and order match CONTEXT.md verbatim — **do not rename or reorder**:

| Position | Card label | Source metric | Delta source |
|----------|------------|---------------|--------------|
| 1 (leftmost) | `Total revenue` | `current.revenue` | `(current.revenue - previous.revenue) / previous.revenue` |
| 2 | `Avg sell-through` | `current.sell_through` (already a ratio 0–1) | `current.sell_through - previous.sell_through` — **delta is absolute percentage points for ratios, not relative %**. See "Delta semantics" below. |
| 3 | `Total lots sold` | `current.lots_sold` | `(current.lots_sold - previous.lots_sold) / previous.lots_sold` |
| 4 (rightmost) | `Total sales count` | `current.sales_count` | `(current.sales_count - previous.sales_count) / previous.sales_count` |

**Delta semantics (copy-locked so executor doesn't invent variants):**

- **Currency and count deltas** (cards 1, 3, 4) use **relative percentage change**: `(current - previous) / previous`. Rendered as `▲ 12.4%` / `▼ 8.1%` / `—` (no baseline).
- **Sell-through delta** (card 2) uses **absolute percentage-point change**: `current - previous` (both are already ratios 0–1). Rendered as `▲ 3.2pp` / `▼ 1.8pp` / `—`. The `pp` suffix (percentage-point) is the standard financial convention and disambiguates "went from 68% to 71%" (3pp absolute) from "revenue grew 12.4%" (relative). This is a one-card exception with explicit copy — NOT a general rule to apply elsewhere.
- **No-baseline case** (`previous = 0` for cards 1/3/4, or previous sell-through was null/undefined): render a single `—` in muted gray, no suffix, no percentage.
- **Delta suffix** (all cards): `vs previous {periodLabel}` where periodLabel is `YTD` / `6mo` / `12mo` depending on selected period. Rendered small (`text-sm text-gray-500`) on the same line. Full example: `▲ 12.4% vs previous 12mo`.

| Delta state | Full rendered copy |
|-------------|---------------------|
| Positive (currency/count) | `▲ {pct}% vs previous {periodLabel}` |
| Negative (currency/count) | `▼ {pct}% vs previous {periodLabel}` |
| Positive (sell-through) | `▲ {pp}pp vs previous {periodLabel}` |
| Negative (sell-through) | `▼ {pp}pp vs previous {periodLabel}` |
| No baseline | `—` (single em-dash, no suffix) |

`periodLabel` mapping: `ytd → YTD`, `l6m → 6mo`, `l12m → 12mo` (shortened for the inline suffix).

**Loading patterns:**

- **Initial page load (both queries pending):** Render the page shell (heading + period selector skeleton placeholder in the same 40px height so layout is stable), then 4 `KpiCardSkeleton` tiles in the KPI grid, then the `Recent sales` section heading, then 5 `RecentSaleCardSkeleton` tiles. Show the skeletons regardless of which query completes first — this avoids the landing page flashing from "half-loaded" to "fully-loaded" twice. Total skeleton count: 4 KPIs + 5 recent sales = 9 shimmer elements.
- **Period change (re-fetch with existing data):** Do NOT replace cards with skeletons. Apply a subtle 2px top border-loading indicator on the KPI section container (`border-t-2 border-accent animate-pulse motion-safe:animate-pulse` for the duration of the in-flight query). Recent Sales is not affected by period changes (it always shows the 5 most recent regardless of period). 2px is `border-t-2` — a Tailwind-native multiple-of-2, acceptable here because borders are CSS border-width not layout spacing; the spacing-scale token rule applies to layout gaps/padding/margins.
- **Period-selector button press while loading:** Button remains enabled (rapid period toggling is a valid UX — TanStack Query dedupes/cancels in-flight requests). The 2px border indicator on the KPI section is the only motion cue.
- **Recent Sales card skeleton (`RecentSaleCardSkeleton`):** Same card shape as `RecentSaleCard`, with `h-3` bars for date/sell-through rows and `h-4` bars for sale-number/title/net-revenue rows. Shimmer via `motion-safe:animate-pulse` on `bg-gray-200 dark:bg-gray-700` bars — identical pattern to Phase 3 `TableSkeleton`.
- **KPI card skeleton (`KpiCardSkeleton`):** Same card shape as `KpiCard`, with `h-3 w-24` bar for the label row, `h-4 w-32` bar for the value row, `h-3 w-40` bar for the delta row. All inside a `p-6 rounded-lg border border-gray-200 dark:border-gray-700 min-h-[128px] space-y-2` container — so the skeleton's dimensions match the final card 1:1, eliminating layout shift on data arrival.

**Error patterns:**

- Each section (KPIs, Recent Sales) fails **independently** per CONTEXT.md. If `useKpiSummary` errors, render `<ErrorState heading="Couldn't load KPIs" body="..." onRetry={refetchKpi} />` in the KPI section and leave Recent Sales alone (or vice versa).
- Inline error states only. No toasts. No modal error dialogs.
- Every error heading has `role="alert"` — matches Phase 1 and Phase 3.
- Retry is always `Retry` — single verb. Same Phase 3 intentional-consistency exception applies; do not introduce `Try again` or `Refresh`.

**Keyboard tab order on `/` (Dashboard):** header user menu (inherited shell) → period selector (first focusable option, arrow-keys to navigate within) → past selector to Recent Sales first card link → next card → ... → last card. KPI cards are skipped (not focusable). If KPI section is in error state, the Retry button inserts between period selector and first Recent Sales card.

### Responsive breakpoints

- **Default (desktop, 1024px+):** KPI grid 4 columns, Recent Sales grid 5 columns, period selector inline to the right of the page heading.

- **Below 768px:** KPI grid collapses to 1 column (4 stacked cards), Recent Sales grid collapses to 1 column (5 stacked cards). Per INFR-03, below-768px is out of scope for detailed optimization — `grid-cols-1` as the mobile fallback ensures the page is usable but not mobile-polished.

## Accessibility Floor

Non-negotiable — the checker flags any absence.

- **Focus indicators:** PeriodSelector options, Recent Sales cards, Retry buttons all have `focus-visible:ring-2 focus-visible:ring-accent focus-visible:ring-inset`. KPI cards are not focusable (not interactive) — no focus indicator required.
- **Delta direction announced non-visually:** Each KPI delta span has an `aria-label` in the form `Up {text} versus previous {periodLabel}` / `Down {text} versus previous {periodLabel}` / `No baseline comparison`. This ensures screen readers convey direction without relying on the `▲`/`▼` glyph (some assistive tech skips non-BMP glyph pronunciation).
- **Period selector pattern:** WAI-ARIA radiogroup — `<fieldset aria-label="...">` wrapping `<button role="radio" aria-checked={...}>` children. Roving `tabIndex` so Tab moves past the whole selector, arrow keys move within it. Active option has `aria-checked="true"` and `aria-pressed="true"` (both — broadens AT compatibility).
- **Section headings:** Page uses `<h1>Dashboard</h1>` (inherited rhythm — matches Phase 3 `<h1>Sales</h1>`). KPI section uses `<h2 className="sr-only">Performance summary</h2>` — visually hidden but anchors the ARIA landmark via `aria-labelledby`. Recent Sales section uses visible `<h2>Recent sales</h2>`.
- **Empty / error headings:** Use `role="alert"` on the heading element so screen readers announce state changes (inherited Phase 3 pattern).
- **Color contrast (all verified against WCAG AA 4.5:1 minimum for normal text):**
  - `text-gray-700` on `bg-white` = 10.4:1 (AAA)
  - `text-gray-500` on `bg-white` = 4.6:1 (AA)
  - `text-gray-500` on `bg-gray-50` = 4.5:1 (AA — at floor)
  - `text-gray-900` on `bg-gray-50` = 16.4:1 (AAA) — period-selector active option
  - `text-green-600` (`#16a34a`) on `bg-white` = 4.5:1 (AA — at floor — delta positive)
  - `text-red-600` on `bg-white` = 4.5:1 (AA — delta negative)
  - Accent `#2563eb` on `bg-white` = 4.55:1 (AA — focus rings, inherited)
- **Touch targets:** Period selector option height 40px (WCAG 24px floor satisfied; 40px matches Phase 3 filter input — consistent secondary-control hit area). Recent Sales card min-height 128px (far exceeds 44px clickable-row target). KPI cards not interactive — no target size concern.
- **`prefers-reduced-motion`:** Skeleton shimmer animations use `motion-safe:animate-pulse`. The 2px refetch border indicator on KPI section also uses `motion-safe:animate-pulse`. Both suppress motion when the user prefers reduced motion.
- **Tabular numerics:** KPI values, deltas, Recent Sales card monetary values all use `tabular-nums` — aids users with low vision by aligning digit columns across cards (vertical scanning of the 4-KPI row becomes predictable).
- **Title tooltip fallback:** Recent Sales card titles truncate with ellipsis; native `title` attribute surfaces the full text on hover for both sighted and keyboard users (browsers announce `title` on focus for many AT configurations).
- **Semantic HTML:** Page uses `<main>` (inherited), `<header>`, `<section>` with `aria-labelledby`, `<h1>`/`<h2>` in correct hierarchy, `<fieldset>` for the radiogroup, `<button>` for the period options (NOT `<div>`-as-button), `<a>` (via react-router `<Link>`) for card navigation.
- **Keyboard-only journey:** Tab to period selector → arrow-keys to switch period → Tab past to first Recent Sales card → Arrow-Tab through recent sales → Enter to navigate to a sale detail. No keyboard trap, no unreachable interactive element.
