# Phase 1 — UI Design Contract

> Visual and interaction contract for Foundation & Auth. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Surfaces in scope:**
1. `/login` — email/password login page
2. `/` — Dashboard placeholder shell (sidebar + header + empty content slot)
3. Access-denied state — authenticated non-admin user lands here with sign-out
4. Auth-loading state — brief spinner while session is resolving

**Design north star:** Visually align with TPC App's `Login.tsx` and Tailwind token set so the two apps feel like siblings. Desktop-first (INFR-03) — dashboard uses a left sidebar, not bottom tabs like TPC App.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (raw Tailwind v4) |
| Preset | not applicable |
| Component library | none (hand-rolled; revisit at Phase 3 when tables arrive) |
| Icon library | Heroicons v2 (outline, stroke-width 1.5) — inline SVG, no package dep required |
| Font | System font stack (Tailwind default: `ui-sans-serif, system-ui, …`). Matches TPC App exactly. No custom `@font-face`. |

**Rationale for no shadcn in Phase 1:** CONTEXT.md decision locks visual alignment with TPC App, which uses raw Tailwind with no component library. Phase 1 has only 4 surfaces with minimal interactive complexity (1 form, 1 shell, 1 error screen, 1 spinner). Adopting shadcn now would introduce a second visual vocabulary on day one. Re-evaluate at Phase 3 (data tables) or Phase 4 (KPI cards) when headless primitives start paying for themselves.

**Tailwind theme tokens** (added to `src/index.css` via `@theme` block, mirroring TPC App's pattern):

```css
@import "tailwindcss";

@theme {
  --color-accent: #2563eb;        /* blue-600 — primary CTA, focus rings, active nav */
  --color-accent-hover: #1d4ed8;  /* blue-700 — CTA hover */
}
```

All other colors resolve through Tailwind's default palette (`gray-*`, `red-*`, `white`). Dark-mode classes (`dark:*`) are authored but Phase 1 does not expose a theme switcher — the dashboard follows the OS preference, same as TPC App.

## Spacing Scale

Declared values (multiples of 4 only — Tailwind's default 4px base unit is used as-is):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`space-1`) | Icon-to-label gap, tight inline spacing |
| sm | 8px (`space-2`) | Form field internal gaps, badge padding |
| md | 16px (`space-4`) | Default element spacing, form row gap, button-to-error gap |
| lg | 24px (`space-6`) | Section padding, card padding-block, sidebar link gap, field-to-button gap on login |
| xl | 32px (`space-8`) | Major vertical rhythm (login header-to-form), shell gutters |
| 2xl | 48px (`space-12`) | Page-top spacing on content routes, shell sidebar width (`w-60` = 240px — actual width uses lg tier below) |
| 3xl | 64px (`space-16`) | Reserved; not used in Phase 1 |

**Specific size commitments:**

| Element | Tailwind class | Value |
|---------|----------------|-------|
| Touch targets (buttons, inputs) | `min-h-12` | 48px (matches TPC App) |
| Login card max-width | `max-w-sm` | 384px |
| Login card horizontal margin | `mx-4` | 16px |
| Sidebar width (desktop) | `w-60` | 240px |
| Header height (dashboard shell) | `h-16` | 64px |
| Main content max-width | `max-w-7xl` | 1280px — desktop-first, centered |
| Main content inline padding | `px-8` | 32px |

Exceptions: none. All spacing is a multiple of 4 AND lives in the declared token set above.

## Typography

| Role | Size | Weight | Line Height | Tailwind class |
|------|------|--------|-------------|----------------|
| Body | 16px | 400 (regular) | 1.5 | `text-base` |
| Label | 14px | 600 (semibold) | 1.5 | `text-sm font-semibold` |
| Heading | 20px | 600 (semibold) | 1.3 | `text-xl font-semibold` |
| Display | 24px | 600 (semibold) | 1.25 | `text-2xl font-semibold` |

**Helper/caption copy** (e.g. login subtitle, sidebar "Coming soon" labels, empty-state body): `text-sm` (14px) at weight 400, color `text-gray-500` (light) / `text-gray-400` (dark).

**Weight rule:** exactly two weights — 400 (regular) for body, helper, and error copy; 600 (semibold) for labels, headings, and display. Never introduce 300, 500, 700, 800, or 900 in Phase 1. The `font-medium` class is explicitly out of scope.

**Size rule:** exactly four sizes (14, 16, 20, 24). Error text uses `text-sm` (14px) at weight 400 — same size tier as Label, different weight and color.

## Color

Apply the 60 / 30 / 10 split across every Phase 1 surface.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `white` (`#ffffff`) light / `gray-900` (`#111827`) dark | Page background, main content area, login page surface |
| Secondary (30%) | `gray-50` (`#f9fafb`) light / `gray-800` (`#1f2937`) dark | Sidebar background, header bar, card surfaces |
| Accent (10%) | `#2563eb` (Tailwind `accent` custom token = blue-600) | Primary CTA fill, focus ring, active sidebar-link left border + text |
| Destructive | `red-600` (`#dc2626`) light / `red-400` (`#f87171`) dark | Inline form error text, access-denied heading accent (optional), sign-out button text-only variant |

**Accent reserved for:**
1. The "Sign In" button background on `/login`
2. Focus rings on inputs and buttons (`focus:ring-2 focus:ring-accent`)
3. The active sidebar link's left-edge indicator (2px `border-l-2 border-accent`) and text color
4. The authenticated user's avatar badge in the header (filled circle with initials)
5. The auth-loading spinner border (`border-b-2 border-accent`)

Accent is **not** used for: body links, ordinary buttons, sidebar link hover states (those use `bg-gray-100`), icon defaults, or placeholder content.

**Destructive copy rule:** "Access denied" heading in the non-admin screen stays neutral (`text-gray-900`); only the explanatory error paragraph uses `text-red-600`. This prevents destructive color from becoming "alarming wallpaper."

**Borders:** `border-gray-200` light / `border-gray-700` dark for all dividers (sidebar right edge, header bottom edge, input borders at rest). Input borders shift to `border-accent` on focus.

**Border radius:**
- Inputs, buttons, cards: `rounded-lg` (8px)
- Avatar, spinner: `rounded-full`
- No `rounded-sm`, no `rounded-xl` in Phase 1.

## Interaction Contract

| Interaction | Behavior |
|-------------|----------|
| Login submit (valid admin) | Button replaces its text with a white spinner → navigate to `/` on success |
| Login submit (valid non-admin) | Same spinner → successful auth → `ProtectedRoute` detects `isAdmin === false` → renders access-denied screen |
| Login submit (invalid) | Spinner stops, error paragraph appears below form with `role="alert"`, email field regains focus |
| Input focus | 2px `focus:ring-accent` ring plus `focus:border-accent`; ring and border appear together, no outline offset |
| Sign out (header menu or access-denied screen) | Instant — `supabase.auth.signOut()` → redirect to `/login` |
| Keyboard tab order on login | email → password → Sign In button → (error, if present, announced via `role="alert"`) |
| Sidebar links | Non-interactive placeholders — see copy table |

**Loading patterns:**
- Auth-bootstrapping (before first `onAuthStateChange`): full-page centered spinner, no layout shell yet.
- Inside `/login` after submit: spinner replaces "Sign In" text inside the button; button is `disabled` to prevent double submit.
- Inside `ProtectedRoute` after login while profile loads: same full-page spinner as bootstrapping — reuse the component.

**Error patterns:**
- Inline only. No toasts in Phase 1. No modal error dialogs.
- Every error element has `role="alert"` so screen readers announce it.

## Layout Specifications

### `/login`
- Full viewport height (`h-dvh`) centered both axes.
- Card: `max-w-sm` (384px), centered, no visible border in light mode (the white-on-white card floats on white page — same as TPC App's Login.tsx).
- Vertical stack inside card: title → subtitle → 32px gap → form.
- Inside form: email block → 16px gap → password block → 24px gap → Sign In button → 16px gap → error (when present). The login form uses exactly two gap values drawn from the declared scale: 16px (`space-4`) between same-rhythm elements (field-to-field, button-to-error) and 24px (`space-6`) at the field-to-button transition.

### `/` (Dashboard shell)
- Full viewport, `h-dvh`, CSS grid or flex: left sidebar (`w-60`, full-height) + right content column (flex-1).
- Sidebar vertical order: app title block (48px) → `ANALYTICS` section label (small caps, 24px top padding) → 6 nav links (each 44px tall, 16px padding-inline) → spacer `flex-1` → footer with app version (text-xs, gray-400).
- Right column vertical order: header (`h-16`, sticky, border-bottom) → main content (`flex-1`, overflow-y-auto, `px-8 py-8 max-w-7xl mx-auto`).
- Header content: left side empty (reserved for breadcrumbs in later phases), right side shows `Welcome, {name}` + circular avatar button that opens a 1-item menu ("Sign out").

### Access-denied screen
- Same full-viewport layout as `/login` (centered card), but the card is wider (`max-w-md` = 448px) to fit the longer body text.
- Vertical stack: heading → 8px gap → body paragraph → 24px gap → "Sign out" button (secondary style — outline, not filled accent, because it's a recovery action not a primary action).

## Accessibility Floor

These are non-negotiable; the checker should flag any absence:

- All interactive controls have a visible focus indicator (`focus:ring-2 focus:ring-accent`).
- Every icon-only control has an `aria-label`.
- Error messages use `role="alert"`.
- Form inputs are associated with labels via explicit `htmlFor` / `id`.
- Touch targets are ≥ 48×48px (`min-h-12`) — matches TPC App.
- Color contrast: body text on white ≥ 4.5:1 (gray-900 on white passes), accent-on-white for the CTA passes at 4.5:1 (blue-600 / #2563eb on white = 4.55:1 — confirmed).
- Disabled sidebar links use both `aria-disabled="true"` and a visually muted color so the state is conveyed without relying on color alone.
